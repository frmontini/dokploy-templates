# Psono on Dokploy with PostgreSQL, Cloudflare, and Brevo API

This repository contains a Docker Compose setup for running **Psono Community Edition** on **Dokploy**, behind **Cloudflare**, using a local **PostgreSQL** container and **Brevo API** for transactional email.

It is designed for this flow:

```text
Browser HTTPS -> Cloudflare -> Dokploy/Traefik HTTP -> psono-combo:80
```

No SMTP port is required.

## Files

```text
docker-compose.yml
.env.example
README.md
```

## What this setup does

- Runs Psono using `psono/psono-combo:latest`.
- Runs PostgreSQL locally in a private Docker network.
- Does not expose PostgreSQL publicly.
- Includes a one-shot `psono-keygen` container.
- Allows the first deploy with no environment variables.
- Prints a full copy/paste ENV block in the `psono-keygen` logs.
- Generates the PostgreSQL password locally with Python `secrets`.
- Uses Psono's own `generateserverkeys` command for Psono cryptographic values.
- Prints `PSONO_EMAIL_SECRET_SALT` already escaped with double dollar signs for Dokploy/Compose.
- Keeps the register/backend domain equal to the server domain.
- Supports Brevo through API, not SMTP.
- Does not use `PSONO_SECURE_PROXY_SSL_HEADER`, because that can break Django if passed in the wrong format through env.
- Writes valid physical `config.json` files for the Psono web client.
- Rewrites `config.json` during the first 60 seconds of startup to avoid transient malformed config responses during redeploys.
- Adds a healthcheck that validates `/config.json` as real JSON.
- Adds Traefik HTTP labels for Cloudflare-fronted Dokploy deployments.

## Architecture

```text
Cloudflare
   |
   | HTTPS
   v
Dokploy / Traefik
   |
   | HTTP
   v
psono-combo:80
   |
   +-- nginx web client
   +-- /server -> Psono backend
   |
   v
psono-db:5432
```

## First deploy

Deploy the compose once with no ENV configured.

Expected behavior:

- `psono-keygen` prints a complete ENV block.
- `psono-db` does not start PostgreSQL until `POSTGRES_PASSWORD` exists.
- `psono-combo` waits for the required ENV values.

Open the `psono-keygen` logs and copy the generated ENV block.

Example generated block:

```env
PSONO_DOMAIN=vault.example.com
PSONO_USER_DOMAIN=vault.example.com
PSONO_ALLOWED_DOMAINS=vault.example.com
PSONO_TITLE=Psono Vault
PSONO_ALLOW_REGISTRATION=true
PSONO_ALLOW_LOST_PASSWORD=false

POSTGRES_DB=psono
POSTGRES_USER=psono
POSTGRES_PASSWORD=<generated-password>

PSONO_EMAIL_BACKEND=anymail.backends.brevo.EmailBackend
PSONO_EMAIL_FROM=no-reply@example.com
PSONO_BREVO_API_KEY=REPLACE_WITH_BREVO_API_KEY
PSONO_BREVO_API_URL=https://api.brevo.com/v3/
PSONO_ANYMAIL_IGNORE_UNSUPPORTED_FEATURES=True

PSONO_SECRET_KEY=<generated-value>
PSONO_PRIVATE_KEY=<generated-value>
PSONO_PUBLIC_KEY=<generated-value>
PSONO_ACTIVATION_LINK_SECRET=<generated-value>
PSONO_DB_SECRET=<generated-value>
PSONO_EMAIL_SECRET_SALT=$$2b$$12$$<generated-value>
```

## Domain rule

Keep these three values equal:

```env
PSONO_DOMAIN=vault.example.com
PSONO_USER_DOMAIN=vault.example.com
PSONO_ALLOWED_DOMAINS=vault.example.com
```

Why:

- `PSONO_DOMAIN` is the public vault host.
- `PSONO_USER_DOMAIN` is written into the web client `config.json` as the backend server domain.
- `PSONO_ALLOWED_DOMAINS` controls which domains the backend accepts during registration.

Keeping all three equal avoids client/API domain mismatch problems.

For example:

```env
PSONO_DOMAIN=vault.yourdomain.com
PSONO_USER_DOMAIN=vault.yourdomain.com
PSONO_ALLOWED_DOMAINS=vault.yourdomain.com
```

The generated `config.json` should include:

```json
"domain": "vault.yourdomain.com"
```

Validate:

```bash
curl -s https://vault.yourdomain.com/config.json | python3 -m json.tool
```

## Important: `PSONO_EMAIL_SECRET_SALT`

`PSONO_EMAIL_SECRET_SALT` is a bcrypt-style value generated by Psono. It normally contains dollar signs:

```env
PSONO_EMAIL_SECRET_SALT=$2b$12$...
```

Docker Compose and Dokploy can interpret `$...` as variable interpolation. If pasted raw, the value may be truncated or corrupted.

That is why this compose prints it in the keygen log already escaped:

```env
PSONO_EMAIL_SECRET_SALT=$$2b$$12$$...
```

Copy it exactly as printed into Dokploy.

Inside the running container, it must become:

```env
PSONO_EMAIL_SECRET_SALT=$2b$12$...
```

Validate it after redeploy:

```bash
docker exec -it psono-combo sh -lc 'python3 - <<'"'"'PY'"'"'
import os
v = os.environ.get("PSONO_EMAIL_SECRET_SALT", "")
print("repr:", repr(v))
print("len:", len(v))
print("starts_with_$2b$:", v.startswith("$2b$"))
print("dollar_count:", v.count("$"))
PY'
```

Expected:

```text
starts_with_$2b$: True
dollar_count: 3
```

## Configure real ENV

After copying the generated block, edit placeholders.

Example:

```env
PSONO_DOMAIN=vault.example.com
PSONO_USER_DOMAIN=vault.example.com
PSONO_ALLOWED_DOMAINS=vault.example.com
PSONO_TITLE=Psono Vault
PSONO_ALLOW_REGISTRATION=true
PSONO_ALLOW_LOST_PASSWORD=false

PSONO_EMAIL_BACKEND=anymail.backends.brevo.EmailBackend
PSONO_EMAIL_FROM=no-reply@example.com
PSONO_BREVO_API_KEY=your-real-brevo-api-key
PSONO_BREVO_API_URL=https://api.brevo.com/v3/
PSONO_ANYMAIL_IGNORE_UNSUPPORTED_FEATURES=True
```

Do not manually edit the generated Psono cryptographic keys.

Do not set:

```env
PSONO_SECURE_PROXY_SSL_HEADER=
```

In this Dokploy + Cloudflare HTTP-origin setup, leaving this unset avoids Django errors such as:

```text
The SECURE_PROXY_SSL_HEADER setting must be a tuple containing two values.
```

## Brevo API setup

Use Brevo API, not SMTP.

This matters on providers like DigitalOcean, where outbound SMTP ports may be blocked.

Required ENV:

```env
PSONO_EMAIL_BACKEND=anymail.backends.brevo.EmailBackend
PSONO_EMAIL_FROM=no-reply@example.com
PSONO_BREVO_API_KEY=your-real-brevo-api-key
PSONO_BREVO_API_URL=https://api.brevo.com/v3/
PSONO_ANYMAIL_IGNORE_UNSUPPORTED_FEATURES=True
```

Make sure `PSONO_EMAIL_FROM` is a verified sender in Brevo.

Test email delivery:

```bash
docker exec -it psono-combo \
  python3 ./psono/manage.py sendtestmail your-email@example.com
```

If this fails, do not debug the web client first. Fix email delivery first because registration sends an activation email.

## Cloudflare

Recommended flow:

```text
Browser HTTPS -> Cloudflare -> HTTP origin -> Dokploy/Traefik -> psono-combo:80
```

If Dokploy is not terminating HTTPS, use a Cloudflare SSL/TLS mode that supports HTTP origin, such as Flexible.

## Dokploy and Traefik

The compose includes labels for HTTP routing through Traefik:

```text
vault.example.com -> psono-combo:80
```

The `psono-combo` service is connected to:

```yaml
dokploy-network:
  external: true
```

If your Dokploy Traefik network has another name, update:

```yaml
networks:
  dokploy-network:
    external: true
```

and this label:

```yaml
traefik.docker.network=dokploy-network
```

Check networks:

```bash
docker network ls
```

## `config.json` handling

The Psono web client depends on a valid `config.json`.

This compose writes physical config files before starting the combo process:

```text
/usr/share/nginx/html/config.json
/usr/share/nginx/html/portal/config.json
```

It also rewrites them during the first 60 seconds of startup to avoid malformed responses during redeploys.

Validate public JSON:

```bash
curl -s https://vault.example.com/config.json | python3 -m json.tool
```

If this command formats JSON correctly, the server-side config is valid.

If the browser still says the config is malformed:

- wait a few seconds after redeploy;
- open an incognito/private tab;
- clear browser site data for the vault domain.

## Troubleshooting

### Registration returns HTTP 500

Check logs:

```bash
docker logs --tail=500 psono-combo
```

Search for traceback lines:

```bash
docker logs --tail=500 psono-combo 2>&1 | grep -i -A25 -B10 -E 'traceback|anymail|brevo|email|sender|api|forbidden|unauthorized|invalid|exception'
```

Common causes:

- `PSONO_EMAIL_SECRET_SALT` was pasted raw instead of escaped.
- `PSONO_EMAIL_FROM` is not verified in Brevo.
- Brevo API key is invalid.
- `PSONO_SECURE_PROXY_SSL_HEADER` was set incorrectly.
- Brevo transactional email is not enabled.
- `PSONO_USER_DOMAIN` and `PSONO_ALLOWED_DOMAINS` do not match `PSONO_DOMAIN`.

### `EMAIL_SECRET_SALT incorrect`

Check what the container sees:

```bash
docker exec -it psono-combo sh -lc 'python3 - <<'"'"'PY'"'"'
import os
v = os.environ.get("PSONO_EMAIL_SECRET_SALT", "")
print("repr:", repr(v))
print("len:", len(v))
print("starts_with_$2b$:", v.startswith("$2b$"))
print("dollar_count:", v.count("$"))
PY'
```

Expected:

```text
starts_with_$2b$: True
dollar_count: 3
```

If `dollar_count` is not `3`, paste the value exactly as printed by `psono-keygen`, with double dollars:

```env
PSONO_EMAIL_SECRET_SALT=$$2b$$12$$...
```

### `404 Not Found` with `Content-Type: text/plain`

This usually means Traefik did not match any router.

Check:

```bash
docker inspect psono-combo --format '{{json .Config.Labels}}' | python3 -m json.tool
docker inspect psono-combo --format '{{json .NetworkSettings.Networks}}' | python3 -m json.tool
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Networks}}' | grep -i -E 'traefik|dokploy|psono'
```

### `404 Not Found` with `Server: daphne`

This means the route is pointing to the Psono backend on port `8000`.

That is wrong for the web client.

The public route must point to:

```text
psono-combo:80
```

The `/server/...` backend paths are handled behind the same nginx service.

### Server signature changed warning

If you wiped the database or regenerated Psono keys, the browser may warn that the server signature changed.

That is expected only after a clean reinstall.

Do not ignore this warning on a production vault unless you intentionally regenerated the instance.

## Closing registration

After creating your first user, close public registration:

```env
PSONO_ALLOW_REGISTRATION=false
```

Then redeploy.

## Backup recommendation

Back up at least:

- the PostgreSQL volume;
- the ENV values generated by `psono-keygen`;
- this Docker Compose file.

Without both the database and the original Psono key ENV values, encrypted vault data may become unrecoverable.

## Clean rebuild notes

For a clean rebuild before storing real data:

```bash
docker rm -f psono-combo psono-db psono-keygen 2>/dev/null || true
docker volume ls | grep -i psono
docker volume rm <psono_pg_data_volume_name>
```

Then remove the generated ENV values from Dokploy and redeploy so `psono-keygen` can print a new set.
